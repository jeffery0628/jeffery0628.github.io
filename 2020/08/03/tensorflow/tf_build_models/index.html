tensorflow2 构建模型的三种方式

更新于 2020-08-04 分类于 技术/tensorflow 阅读次数： 评论：
本文字数： 7k 阅读时长 ≈ 6 分钟

缘起

最近想搭建一个tensorflow template，以便快速实现自己的想法。但是tensorflow api 实在太多，太乱。不得不细细梳理一下，以便写出低耦合的框架。

三种定义模型的方式及优先级

keras和tensorflow2.0提供了三种定义模型的方式：

  1. Sequential API
  2. Functional API
  3. Subclassing API

这就需要我们了解如何使用这三种方法来定义模型，还要学会什么时候该用什么方法。上述编号号越小，优先级越高：即先考虑能不能用Keras Sequential APIs 方式建立需要的模型，如果不能，在考虑能不能使用Keras Functional APIs方式来建立你需要的模型，如果还是不能，最后再考虑Keras Model Subclassing APIs方式。

三种模型定义方式

Sequential API

序列模型是layer-by-layer的，它是最简单的定义模型的方法但是有几个不足：

  • 不能够共享某一层
  • 不能有多个分支
  • 不能多个输入
  • 不能多个输出

这种结构的经典网络比如有：Lenet5，AlexNet，VGGNet

Functional API

TensorFlow 2.0提供了另一个符号模型构建api：Keras Functional。有向无环图式的网络结构：

函数API是一种创建更灵活模型的方法。它可以处理非线性拓扑、具有共享层的模型和具有多个输入或输出的模型。：

  • 定义更复杂的模型
  • 支持多输入多输出
  • 可以定义模型分支，比如inception block,resnet block
  • 方便layer共享

Subclassing API

第三种方也是最后一种方法，子类方法subclassing方法（面向对象编程）。在keras中Model类做为基本的类，可以在些基础上，进行会任意个人设置，带来很强的自由。但同时带来的不足就是没用序列和函数定义模型使用起来简单。这对一些研究人员很好，可以对模型部分进行控制。

Keras 模型以类的形式呈现，可以通过继承 tf.keras.Model 这个 Python 类来定义自己的模型。在继承类中，我们需要重写 __init__()（构造函数，初始化）和 call(input) （模型调用）两个方法，同时也可以根据需要增加自定义的方法。

继承 tf.keras.Model 后，可以使用父类的若干方法和属性，如在实例化类 model = Model() 后，可以通过 model.variables 这一属性直接获得模型中的所有变量，免去我们一个个显式指定变量的麻烦。以下将是用subclassing的方法来实现一个VGGNet.

总结

  1. sequential的方式用来实现一层接着一层的模型
  2. functional的模型能完成大多数模型
  3. subclassing的模型能完全自定义，但是比较复杂，常归来说，用functional就好了。

TensorFlow 2.0支持这两种样式:

• 如果目标是易用性、低概念开销：使用Keras序列化API或函数API和内置的训练循环。这是解决大多数问题的正确方法。

• 如果优先考虑灵活性和可编程性，Keras子类化是更合适的choice。

符号APIs的好处和局限性

好处

使用符号APIs，模型是类似于图的数据结构，这意味着可以检查或者总结模型（使用keras.utils.plot_model或者简单的使用model.summary()，或者查看图层、权重和形状描述。）同样，当将层连接在一起时，可以运行兼容性检查（在构建模型时和执行之前）

  • 类似于编译器中的类型检查，可以极大地减少开发人员的错误。
  • 大多数调试将在模型定义阶段进行，而不是在执行阶段。你可以保证任何编译的模型都会运行。这使得迭代更快，调试更容易

符号模型提供一致的API。这使得它们易于重用和共享。例如，在迁移学习中，可以访问中间层激活来从现有的模型构建新的模型，如下所示：

符号模型是由数据结构定义的，这使得它们可以自然地复制或克隆。例如，序列化api和函数api提供了model.get_config()、 model.to_json() 、model.save()、clone_model(model)等功能，可以从数据结构中重新创建相同的模型(无需访问用于定义和训练模型的原始代码)。

局限性

当前生成的符号api是最适合开发的有向无环图模型。这在实践中占了大多数用例，尽管有一些特殊的用例不适合这种简洁的抽象，例如，动态网络(如树状rnns)和递归网络。这就是为什么TensorFlow还提供了一种命令式的模型构建API风格(Keras子类化)。你可以使用序列化api和函数api中所有熟悉的层、初始化器和优化器。这两种样式也是完全可互操作的，因此可以混合和匹配(例如,可以将一种模型类型嵌套到另一种模型类型中)。可以将符号模型用作子类模型中的一个层，反之亦然。

命令式APIs的好处和局限性

好处

实现替换库实现的部件(如，层、激活或丢失函数)变得很容易。这对于编程来说是很自然的，也是深入学习深层知识的好方法。使得快速尝试新想法变得容易，并且对研究人员特别有帮助。使用Python，在模型的正向传递中指定任意控制流也很容易。

局限性

在使用命令式API时，模型是由类方法的主体定义的。模型不再是透明的数据结构，而是一段不透明的字节码。

  1. 代码没有了对输入或层间兼容性的检查，所以当使用这种风格时，很多调试负担从框架转移到开发人员身上。

  2. 命令式模型可能难于重用，提取激活的方法是用新的调用(或前向)方法编写一个新类。最初写可能很有趣，也很简单，但如果没有标准，它可能会为以后埋坑。

  3. 命令式模型也更难于检查、复制或克隆。例如，model.save()、model.get_config()和clone_model不能用于子类化的模型。model.summary()只给出一个层列表(不提供关于它们如何连接的信息，因为这是不可访问的)。
